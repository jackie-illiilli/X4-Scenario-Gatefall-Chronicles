# Gatefall Chronicles：战役设计方法与 XML 用法

## 0. 这份文档现在负责什么

这份文档不再讨论“理论上也许能做什么”，而是总结我们已经在 `extensions/gatefall_chronicles` 里完成《任务1：暗流涌动》与《任务2：祸水东引》后，确认可用、值得复用的做法。

它的目标很直接：

- 帮后续战役先把结构设计对，再开始写 XML。
- 让“剧情稿 → 任务流程 → 地图/MD/Cutscene/文本”之间的映射更稳定。
- 提前避开我们在任务1里已经踩过的坑。

任务 2 新增的长流程 cue、地图缩放、站内出生、动态室内、部署物检测、OOS 舰队命令和复杂评分经验，集中记录在 [任务2制作经验与复用清单.md](任务2制作经验与复用清单.md)。本文件保留跨任务的总原则，避免重复维护同一套细节。

如果只看一条原则，那就是：

> 战役的关键结果必须由 MD 保证，Cutscene 负责表现，AI 只负责补气氛，不负责决定剧情是否成立。

## 1. 目前 Gatefall 已验证的文件分工

当前正式实现目录如下：

```text
extensions/gatefall_chronicles/
├─ content.xml
├─ libraries/
│  ├─ gamestarts.xml
│  └─ loadouts.xml
├─ maps/
│  └─ gatefall_scenario_01_galaxy_macro.xml
├─ md/
│  ├─ gatefall_lib_scenario.xml
│  └─ gatefall_scenario_01.xml
├─ cutscenes/
│  ├─ gatefall_s01_phase1_cutscene.xml
│  ├─ gatefall_s01_phase3_cutscene.xml
│  ├─ gatefall_s01_phase5_cutscene.xml
│  ├─ gatefall_s01_phase6_cutscene.xml
│  ├─ gatefall_s01_phase7_cutscene.xml
│  └─ gatefall_s01_phase10_cutscene.xml
└─ t/
   ├─ 0001-L044.xml
   └─ 0001-L086.xml
```

建议后续每一关继续沿用这个结构：

- `gamestarts.xml`：入口、描述、评分阈值。
- `loadouts.xml`：玩家船、关键敌舰、关键友军的固定装备。
- `*_galaxy_macro.xml`：这关自己的地图快照。
- `gatefall_scenario_xx.xml`：主流程、刷船、目标、失败/成功、计分。
- `gatefall_lib_scenario.xml`：跨关通用库，比如结算、统计、公共工具函数。
- `cutscenes/*.xml`：纯分镜和镜头。
- `t/*.xml`：文本、字幕、任务名、描述、目标、结算文本。

## 2. 推荐制作顺序

不要一上来就同时做地图、剧情、镜头、刷船、评分。我们在任务1里证明，这样返工会非常多。

推荐顺序：

1. 先写 `.md` 战役大纲，只写“玩家实际会做什么”。
2. 把大纲拆成 Phase，并明确每个 Phase 的：
   - 进入条件
   - 成功条件
   - 失败条件
   - 过场触发点
   - 需要刷出的对象
3. 再建立 `gamestarts.xml + 地图 + 主 MD` 的最小可运行版本。
4. 先把任务目标、刷船、传送、结算跑通。
5. 然后再补 Cutscene。
6. 最后才做评分、BGM、细对白、命名、字幕时长。

一句话总结：

> 先让任务能完整打完，再让它看起来像一个正式战役。

## 3. 剧情稿怎么写，后面最省事

后续 `.md` 剧情稿，建议固定写成下面的结构：

### 3.1 每个 Phase 必须有四样东西

- 场景目的：这一段想让玩家理解什么。
- 玩家动作：玩家真正要做什么。
- 关键事件：这一段必须发生什么。
- 分镜提示：哪些台词需要切镜头。

### 3.2 不要只写“演出”，一定要写“可执行动作”

比如不要只写：

- “远处出现异常信号，气氛神秘”

应该写成：

- 玩家部署卫星后，40km 外刷出异常信号锚点。
- 台词 `<t id="533">` 触发时播放一次异常信号跃迁特效。
- 镜头切到异常信号的远景固定镜头。

这样后面写 MD 和 Cutscene 时不会猜。

### 3.3 分镜描述尽量和文本 id 绑定

任务1的经验非常明确：

- 如果只写“这里切个镜头”，后期一定会混乱。
- 如果写成“`<t id="533">` 切到异常信号”，“`<t id="534">` 切回玩家”，实现就很稳定。

所以后续文档建议直接写：

- `t id`
- 镜头目标
- 镜头类型
- 持续时间
- 是否需要平移/环绕/固定

## 4. MD、Cutscene、文本三者的正确分工

### 4.1 MD 负责“真实逻辑”

MD 负责：

- 创建/销毁舰船、空间站、残骸、信号源
- 推进任务阶段
- 设置目标
- 计时
- 控制玩家船位置、朝向、锁定状态
- 触发结算
- 写评分统计
- 播放/停止 BGM
- 决定何时播放 Cutscene

### 4.2 Cutscene 负责“镜头表现”

Cutscene 负责：

- 镜头位置
- 镜头运动
- 镜头切换
- 展示哪个对象

Cutscene 不要负责：

- 决定船什么时候刷出来
- 决定任务什么时候完成
- 决定对象是否死亡

这些都应该由 MD 完成。

### 4.3 文本文件只负责“内容映射”

`0001-L086.xml` / `0001-L044.xml` 应只负责：

- 对话内容
- 任务名
- 目标文字
- 结算文字
- 说话人名称

不要把流程逻辑埋进文本层。

## 5. 我们已经确认好用的战役结构

后续新关卡，建议每关主 MD 都保留下面这套骨架：

```text
Start
├─ Setup
├─ Phase1_Intro
├─ Phase2_ObjectiveA
├─ Phase3_CombatA
├─ Phase4_Travel
├─ Phase5_Setpiece
├─ Phase6_Defence / Chase / Escape
├─ Phase7_FinalWave
├─ Success
├─ Failure
└─ Cleanup
```

经验上，Phase 数可以多，但每个 Phase 的职责必须单一。

最怕的是一个 cue 同时管：

- 刷船
- 播对话
- 切镜头
- 计时
- 成功失败

这种后面一改就全连着坏。

## 6. 地图设计：能少就少，但必须够剧情用

任务1已经验证，Timelines 战役地图没必要做成完整主宇宙。更合理的做法是：

- 只保留这关实际会经过的星区/通道/星门/超级高速路。
- 需要的地方单独布置战场对象。
- 让地图为剧情服务，而不是为“宇宙完整性”服务。

### 6.1 地图最值得优先确认的点

- 玩家出生点是否正确
- 星门/加速器朝向是否正确
- 玩家穿越后的朝向是否正确
- 异常信号、巡逻队、主力舰是否位于可渲染范围
- 关键空间站和援军是否不会被大型结构挡住

### 6.2 任务1验证出的渲染经验

镜头想拍到远处舰船时，不要只看逻辑距离，还要考虑渲染距离。

目前实战经验：

- 40km 左右内更容易稳定看见中小型船。
- 30km 左右内更适合拍成群的大型舰。
- 太远虽然逻辑存在，但镜头里不一定渲染出来。

所以“震撼感”更多靠：

- 把舰队刷得更密
- 镜头拉远
- 分波跃迁入场

而不是单纯把距离拉到 60km 以上。

## 7. 刷船原则：关键对象固定，杂兵可模板化

### 7.1 关键对象必须固定

以下对象建议固定宏、固定装备、固定位置、固定朝向：

- 玩家船
- 僚机
- 关键敌舰
- 关键友军旗舰
- 剧情镜头主角舰

原因很简单：一旦这些对象随机化，镜头、难度、剧情演出都会飘。

### 7.2 杂兵可以模板化

普通波次敌人可以按“舰种模板 + 数量”做批量创建。

但即便如此，也建议把以下信息写死在脚本侧：

- 刷新位置类型
- 面向目标
- 首条命令
- 是否追击玩家
- 是否以空间站为主目标

### 7.3 追兵不要只靠“看起来会追”

任务1已经证明，跨星区追兵如果只是“自然 AI 追击”，经常会：

- 不过门
- 不走高速路
- 丢失目标
- 到了另一个星区后不继续施压

推荐做法：

- 如果剧情必须追上，就显式指定 `pursuetargets` 或明确追加命令。
- 如果还是不稳定，就在目标星区出口补刷追兵。

一句话：

> 能脚本保证的追击，就不要把关键压迫感交给自然 AI。

## 8. Cutscene 设计经验：宁可稳，也别乱转

任务1里最明显的一条经验就是：

> 快速旋转镜头很容易看起来廉价，而且玩家更容易晕。

后续镜头建议优先使用这几类：

- 固定远景
- 固定近景
- 缓慢环绕
- 缓慢平移
- 从玩家视角附近接管

尽量少用：

- 高速绕圈
- 很短时间内大角度切换前后方
- 目标距离很远的长距离飞镜

### 8.1 镜头切换最好绑定台词节点

推荐模式：

- 某句台词开始时切镜头
- 下一句台词开始或结束时结束镜头

这样不会出现“对白结束了，镜头还在飘”的问题。

### 8.2 过场中的对象控制

如果玩家在 Cutscene 时还能乱飞，通常会破坏演出。

任务1已经验证，涉及传送、异常信号、强制观战时，可以：

- 暂停玩家动作
- 锁定玩家
- Cutscene 完成后再解锁

## 9. 特效经验：视觉表现优先用“可确认显示”的方案

任务1里我们已经确认：

- 开场星门类特效可用。
- HQ 跃迁特效理论可用，但不是所有触发时机都稳定可见。
- 某些“加了但没看到”的问题，本质不是 XML 写错，而是对象、时机、镜头、渲染范围没有对齐。

所以后续处理特效时，建议按这个顺序排查：

1. 对象是否真实存在且可见
2. 特效是否挂在正确对象上
3. 镜头是否拍到
4. 特效播放时对象是否已被移动/销毁
5. 是否被别的结构遮挡

不要默认“特效代码写了就一定能被玩家看到”。

## 10. BGM 的推荐用法

### 10.1 环境 BGM 和剧情 BGM 分开想

我们已经确认，一个战役里至少有两种音乐逻辑：

- 环境常驻 BGM
- 某一阶段临时强制 BGM

任务1里比较合适的方式是：

- 阿尔特弥斯的朦胧阶段单独指定一首氛围曲
- 进入战斗或结尾援军阶段再切到剧情战斗曲
- 某段结束后恢复自动战斗切换逻辑，或者显式停掉强制曲

### 10.2 不要长期把整关锁死在一首曲子上

实战里会出现一个问题：

- 一旦你一直强制播放固定曲目，原版的战斗自动切歌逻辑就不工作了。

所以建议：

- 只在真正需要控场的阶段手动播 BGM
- 其余时间尽量回到环境音乐或默认战斗逻辑

### 10.3 自定义 BGM 的文件结构与注册方式

自定义音乐不要放进或修改官方 `ego_dlc_timelines` 目录。统一放在本扩展内，例如：

```text
extensions/gatefall_chronicles/
├─ libraries/
│  └─ sound_library.xml
└─ music/
   ├─ mission_01/
   │  ├─ briefing.ogg
   │  ├─ tension_01.ogg
   │  └─ battle.ogg
   └─ common/
      ├─ victory.ogg
      └─ failure.ogg
```

推荐使用 OGG Vorbis，文件名尽量只用英文小写、数字和下划线。XML 中引用音乐文件时通常省略 `.ogg` 后缀。

单首无限循环 BGM 在 `libraries/sound_library.xml` 中可注册为：

```xml
<?xml version="1.0" encoding="utf-8"?>
<diff>
  <add sel="/soundlibrary">
    <sound id="gfc_music_m01_battle"
           description="Gatefall mission 01 battle music"
           repeat="-1" is3d="0" preload="0">
      <sample start="extensions\gatefall_chronicles\music\mission_01\battle"/>
      <volume start="0.70"/>
      <effects>
        <fade in="1.5" out="1.0"/>
      </effects>
    </sound>
  </add>
</diff>
```

约定：

- ID 使用全局唯一的 `gfc_music_` 前缀。
- `repeat="-1"` 表示无限循环。
- `is3d="0"` 表示非空间化背景音乐。
- `preload="0"` 避免提前加载大型音乐文件。
- `volume` 和 `fade` 应按战役实际听感调整。

### 10.4 多首 BGM 随机集合

同一阶段需要每次游玩随机选曲时，可以注册一个带权重的集合：

```xml
<sound id="gfc_music_m01_tension_pool"
       description="Gatefall mission 01 tension pool"
       repeat="-1" is3d="0" preload="0">
  <sample selection="random">
    <select weight="4" start="extensions\gatefall_chronicles\music\mission_01\tension_01"/>
    <select weight="3" start="extensions\gatefall_chronicles\music\mission_01\tension_02"/>
    <select weight="1" start="extensions\gatefall_chronicles\music\common\dark_ambient"/>
  </sample>
  <volume start="0.60"/>
  <effects>
    <fade in="2.0" out="1.0"/>
  </effects>
</sound>
```

`weight="4/3/1"` 是相对抽取权重。这个结构表示“调用该音乐 ID 时随机选择”，不应当当作能够严格控制曲目先后顺序的传统播放列表。需要按 `briefing → tension → battle → victory` 固定推进时，应把每首音乐注册成独立 ID，再由 MD 的剧情节点负责切换。

### 10.5 MD 播放、切换与清理

无限循环音乐的定义和调用必须对应：

```xml
<!-- sound_library.xml 中 repeat="-1"，MD 中必须声明 loop="true" -->
<play_music music="'gfc_music_m01_battle'" loop="true"/>
```

切换音乐时先停止上一首：

```xml
<stop_music/>
<play_music music="'gfc_music_m01_climax'" loop="true"/>
```

任务胜利、失败、玩家死亡、主动放弃、超时、场景退出和 cue 清理路径都必须考虑执行：

```xml
<stop_music/>
```

否则无限循环的任务音乐可能继续占用音乐通道，覆盖正常环境音乐。最好把这项操作放进战役统一的 `Cleanup` 或 `EndScenario` 流程。

只播放一次的过场或结算音乐可使用 `repeat="1"`，调用时不写 `loop="true"`：

```xml
<sound id="gfc_music_m01_victory" repeat="1" is3d="0" preload="0">
  <sample start="extensions\gatefall_chronicles\music\common\victory"/>
</sound>
```

```xml
<play_music music="'gfc_music_m01_victory'"/>
```

### 10.6 不能指定从曲目的某个时间点开始播放

当前确认的 X4 标准音乐接口不支持 seek。MD 的 `<play_music>` 只有音乐 ID 和循环声明，没有 `offset`、`position`、`seek` 或音乐播放用的 `starttime` 参数：

```xml
<play_music music="'gfc_music_example'" loop="true"/>
```

`sound_library.xml` 中的：

```xml
<sample start="extensions\gatefall_chronicles\music\mission_01\example"/>
```

这里的 `start` 是音频资源路径，不是开始播放时间。因此不能通过 XML 要求“从 2 分 35 秒开始播放”。在 MD 中等待 155 秒后再调用，也只会让音乐晚 155 秒开始，文件仍从 `00:00` 播放。

需要指定起点时，采用预裁剪文件：

```powershell
ffmpeg -ss 00:02:35 -i "battle_original.ogg" -c:a libvorbis -q:a 6 "battle_from_02m35s.ogg"
```

然后把裁剪后的文件注册为独立 ID，例如：

```text
gfc_music_battle_intro       原曲 00:00 开始
gfc_music_battle_tension     原曲 01:20 开始
gfc_music_battle_climax      原曲 03:45 开始
gfc_music_battle_finale      原曲 05:10 开始
```

如果要求不同片段之间无缝衔接，应在音频编辑阶段预先处理切入点、节拍、淡入淡出和循环边界；MD 本身不能在同一个音频流中跳转播放位置。

## 11. 字幕与对话系统经验

### 11.1 “字幕系统”比系统提示更适合剧情

任务1已经确认：

- 玩家、僚机、舰载 AI 的交流，统一走字幕对话体验更好。
- 不要混用“系统提示一句、对白一句”的方式。

### 11.2 说话人命名尽量短、稳定

已验证比较好用的命名方式：

- 玩家
- 僚机
- 舰载AI
- 空间站管理员
- 指挥部 / 指挥舰名称

不要在同一关里频繁改称呼，否则玩家会花精力辨认是谁在说话。

### 11.3 文本 id 规划建议

建议继续保持：

- `1-99`：标题、描述、结算标题
- `100-299`：任务目标、加载文本
- `500+`：对白
- `1000+`：结算说明

这样后面维护多语言会轻松很多。

## 12. 评分系统：能稳定统计的才纳入评分

这是任务1里最值得写进方法论的一条。

### 12.1 推荐优先纳入评分的项

这些通常更稳定：

- 玩家最终结构
- 友方关键空间站最终结构
- 是否完成可选目标
- 是否超时
- 是否触发特殊失败

### 12.2 对“玩家亲自击杀数”要非常谨慎

任务1的经验是：

- “玩家亲自击杀多少 M / S”这个统计，实战里不一定稳定。
- 尤其是混战、防守战、友军援军很多时，归属判定容易出问题。

所以后续推荐顺序是：

1. 先尝试做，并单独验证。
2. 如果验证不稳，就不要把它当核心评分项。
3. 宁可改成“总防守表现分”，也不要保留一个经常统计失败的指标。

也就是说：

> 评分系统最重要的不是精致，而是可解释、可复现。

### 12.3 星级映射建议

推荐继续采用：

- 100-80：5星
- 80-60：4星
- 60-40：3星
- 40-20：2星
- 20-5：1星
- 5 以下：0星

但每关最好保证：

- 正常通关至少能拿到 1-2 星
- 熟练游玩能稳定拿到 3-4 星
- 5 星需要明显更优表现

## 13. 结算与成功判定

### 13.1 任务完成不要只看“最后一句台词播完”

任务1已经遇到过：

- 结算比台词早
- 还没播完结尾对白就进入结算

后续建议成功流程明确拆成：

1. 最终目标达成
2. 播尾声对白
3. 如有需要，播放最后一个镜头
4. 延迟数秒
5. 进入结算

### 13.2 最终清剿阶段比“援军一来就结算”更完整

任务1里后期改成：

- 援军到场
- 给玩家一个“清掉附近 Xenon”的最后目标
- 再进入尾声和结算

这个结构比“援军一来就结束”更像正式战役。

后续值得复用。

## 14. 后续新战役推荐沿用的工程规则

### 14.1 一个战役，一份主 MD

不要把多关主流程混在一个文件里。建议：

- `gatefall_scenario_01.xml`
- `gatefall_scenario_02.xml`
- `gatefall_scenario_03.xml`

公共逻辑才放进 `gatefall_lib_scenario.xml`。

### 14.2 一个大段演出，一个 cutscene 文件

任务1按 phase 切分 cutscene 是对的，后续继续沿用。

优点：

- 好定位
- 好回退
- 不会一个文件越改越乱

### 14.3 地图不要提前做“全战役版”

每一关先做自己的地图快照，能跑通后再考虑共用地图资源。

原因是：

- 关卡剧情会不断改
- 星门、加速器、空间站、敌军分布会反复调整
- 先做总地图，后面只会增加维护成本

## 15. 后续写下一关前，建议先过一遍这个检查表

### 设计层

- 玩家动作是否明确
- 每个 Phase 的成功/失败条件是否明确
- 分镜是否绑定到具体台词 id
- 关键对象是否都写明了“谁、在哪、什么时候出现”

### 脚本层

- 关键事件是否由 MD 保证
- 玩家是否可能在过场里破坏演出
- 需要追击的敌人是否有明确追击逻辑
- 目标切换是否和阶段切换同步

### 演出层

- 镜头是否过快
- 关键对象是否在可渲染距离内
- 特效是否真的会被看见
- 尾声对白是否会在结算前播完

### 评分层

- 评分项是否稳定可统计
- 失败后是否能解释为什么低分
- 五星是否明显高于普通通关表现

## 16. 当前最值得保留的核心经验

最后把这次任务1真正沉淀下来的经验压缩成几句：

1. 先做“能打完”的任务，再做“好看”的任务。
2. 关键剧情永远交给 MD，不交给自然 AI。
3. Cutscene 负责镜头，不负责结果。
4. 关键对象固定化，杂兵模板化。
5. 评分只统计稳定拿得到的数据。
6. 镜头宁可稳，也别乱转。
7. 地图只做剧情真正需要的部分。

后续如果新战役又验证出新的稳定做法，就继续把结论补回这份文档，而不是只留在对话记录里。
