# 任务2制作经验与复用清单

本文记录《任务2：祸水东引》在实际落地过程中验证过、踩过坑、后续可直接复用的做法。  
建议将本文视为“任务2级别复杂度战役”的补充手册；通用原则仍以《战役设计方法与XML用法.md》为主。

## 1. 这份清单主要解决什么问题

任务 2 和任务 1 最大的不同，不在于“敌人更多”，而在于系统耦合明显变多：

- 玩家不再总是出生在舰船上，而是可能出生在空间站室内。
- 任务目标不再是单段战斗，而是“地图操作 -> 护航/防守 -> 室内对话 -> 部署物 -> 潜入数据库 -> 跨星区布置”的长链流程。
- 同一批舰船、民船、空间站会跨多个 Phase 持续存在，后续还要计分。
- 玩家不一定亲自执行部署动作，可能要改成队长机或其他 NPC 完成。
- 需要同时处理：字幕、镜头、BGM、计时器、动态室内、电梯权限、地图提示、任务标记、部署检测、关系移交、评分统计。

因此，任务 2 的经验重点不是“某个 XML 标签怎么写”，而是“哪些东西必须拆开、哪些东西不能混在一个 cue 里”。

## 2. 长流程任务的核心原则：一个 cue 只做一件事

任务 2 最容易出问题的地方，就是把下面几类事情塞进同一个 cue：

- 播对话
- 切任务目标
- 刷敌人
- 给民船/矿船下移动命令
- 开计时器
- 开启下一段对话

实际经验表明，只要这些动作混在一起，最容易出现以下问题：

- 对话还没播完，下一阶段目标已经提前跳了。
- 某个 `<signal_cue_instantly>` 被别的分支抢先触发，导致 521/522/541/542 这种对话断链。
- 民船移动命令和刷敌命令绑定在一起时，只要其中一边条件没满足，两边都可能失效。
- 倒计时阶段结束了，但清场 cue 还没进，后续剧情提前开始。

更稳的写法是拆成四层：

1. 对话 cue：只负责字幕、镜头、锁控、BGM。
2. 任务 cue：只负责 `set_objective`、`show_help`、倒计时、任务标记。
3. 刷新 cue：只负责 `create_ship` / `create_group` / `create_station`。
4. 命令 cue：只负责给已有对象下命令、改关系、开权限、记录状态。

如果一个 Phase 里确实要连续触发多件事，也尽量串成：

- `DialogueEnd -> ObjectiveStart`
- `ObjectiveStart -> SpawnWave`
- `SpawnWave -> IssueOrders`

不要把三步揉成同一个 action block。

## 3. 对话时序：一定要用“最后一句结束”来推动阶段

任务 2 已经验证过，最稳的推进方式不是“某句开始时”，而是“最后一句播完后”。

建议：

- 一个 Phase 的任务目标，在该 Phase 的开场说明对话全部结束后再设置。
- 敌人刷新，尽量挂在明确的一句剧情台词之后，而不是前一段对话开始时。
- 如果要给剧情留空白，使用显式延迟，例如“509 结束后 10 秒触发 521”，不要依赖玩家行为恰好补上这段空窗。

特别是下面几种情况，必须显式加保护：

- 对话后延迟触发：用单独 cue 挂 `<delay>`。
- 只触发一次：加布尔状态或“对象已创建”判断。
- 倒计时到时自动转阶段：Phase 完成和 Phase 超时都必须汇总到同一个收口 cue，避免双重触发。

## 4. 守卫/护航任务：超时推进要和正常完成共用收口

任务 2 的 Phase 2、Phase 3 都有一个重要经验：  
“杀完敌人”和“撑满时间”本质上是两种完成路径，但后续只应该进入同一个结算入口。

推荐模式：

- `PhaseX_ObjectiveActive`
- `PhaseX_AllTargetsDead`
- `PhaseX_TimeoutReached`
- `PhaseX_Resolve`

前两个或后三个都只负责发信号，真正进入下一阶段的一定是 `Resolve`。

这样可以避免：

- 清场和超时同时发生，导致下一阶段重复开启。
- 一边还在刷船，另一边已经开始和总指挥说话。
- TEL 货船/HAT 矿船还没领到移动命令，任务就跳段了。

## 5. 地图提示与“打开地图”检测：只把它当成剧情开关

任务 2 已经验证：

- 用“玩家打开地图”作为剧情开关是可行的。
- 但不要把“打开地图”和“关闭地图”都绑在关键流程上，过于脆弱。
- 更适合的用法是：先给系统提示，再监听一次地图打开事件，作为“玩家已经被引导到指挥界面”的确认。

这类触发最适合用来做：

- 开场教学提示
- 某段剧情正式开始
- 引导玩家第一次接触战术地图

不适合直接拿来做：

- 严格的多步任务判定
- 依赖玩家连续 UI 操作的流程

## 6. 室内出生与动态室内：别只建站体，房间权限也要单独管

任务 2 里已经踩过一个典型坑：  
“空间站本身刷出来了，但玩家还是出生在错误位置，或者出生在建模内部。”

要点有三条：

- `gamestarts.xml` 里的出生方式要和任务地图里的站内空间、房间定义一致。
- 如果使用类似 Timelines `battle_1` 的站内出生方案，不能只抄表面对象；要一起核对动态室内、房间宏、入口点和玩家初始 actor 位置。
- 指挥中心、电梯权限、数据库访问权限最好分阶段开放，不要开局一次性全开。

任务 2 的稳妥经验：

- 前期只开放玩家必要的站内活动区域。
- 到了和总指挥对话的 Phase，再开放指挥中心电梯权限。
- 数据库相关电梯权限，不要在拿到密钥前开放。

## 7. 室内对话镜头：固定目标比复杂环绕更稳

室内场景的镜头非常容易穿模，尤其是：

- 环绕半径太大
- 目标点过高
- 房间碰撞体复杂
- 相机在室内绕完整圈

任务 2 实测下来，最稳定的方案是：

- 玩家进入指挥室后先延迟约 1 秒，再开始镜头。
- 优先使用“从前方对准总指挥”的近距离固定镜头。
- 如果要环绕，半径要小、速度要慢，并且要先验证不会穿墙。
- 长对话不要频繁切镜头，宁可整段共用一个稳定镜头。

一句话总结：  
室内镜头首先要“稳且能看清脸”，其次才是“好看”。

## 8. 部署类任务：任务提示物和真正部署物必须分开

任务 2 已经反复验证，部署人造卫星/导航信标这类任务里，最容易误判的是：

- 用来提示位置的信标/卫星
- 玩家或 NPC 真正部署的卫星/导航信标

这两类对象绝对不能混用。

建议固定做法：

- 用一个“只做提示”的对象标记部署点。
- 真正任务完成判定，监听实际发射事件或检测新生成的部署物。
- 判定时明确校验：
  - 是否玩家阵营拥有
  - 是否宏名正确
  - 是否落在指定半径内
  - 是否是本阶段发射的对象

否则就会出现：

- 开局摆的提示信标直接被当成任务完成物。
- 玩家仓库里原本就有的同类对象误判。
- 队长机部署了，任务却还在等玩家部署。

## 9. 队长机代替玩家执行部署：最好做成独立流程

任务 2 已经不是“玩家自己手动放全部东西”，而是部分动作由队长机执行。  
这类流程不要硬套玩家部署逻辑。

更稳的结构是：

- 任务目标告诉玩家“掩护/前往/确认”。
- 队长机拥有对应物品库存。
- 单独 cue 负责让队长机移动到部署点。
- 单独 cue 负责监听或模拟部署完成。
- 完成后再切下一段对话。

要点：

- 队长机库存和玩家库存要分开核对。
- 任务文本也要明确是谁在执行部署。
- 如果部署要计分或影响后续镜头，部署完成对象必须保存到变量里，后面继续复用。

## 10. 刷船后一定要分离“创建”和“下命令”

任务 2 多次出现过这样的问题：

- 舰船刷出来了，但不动。
- 本来要打空间站，结果去追玩家。
- TEL 货船/HAT 矿船应该前往下方星门，却停在原地。

经验上最稳的写法是：

- 创建对象后先显式保存引用。
- 如有必要，先 `cancel_all_orders`。
- 再单独写命令 cue，给它明确的攻击/移动/保护指令。

尤其是这些目标：

- 民船
- 矿船
- 不该乱动的剧情舰
- 需要精确攻击指定目标的 Xenon 波次

不要依赖默认 AI 行为“自己懂”。

## 11. 跨 Phase 保留对象：凡是要计分的都别随便删

任务 2 和任务 1 的一个共通大坑是：  
剧情里为了挪位置、改场景、切镜头，常常想直接删对象再重建。

但只要某对象涉及以下任一用途，就尽量不要这么做：

- 后续任务目标
- 评分对象
- 关系移交对象
- 镜头目标
- 对话发言者

尤其是：

- 队长机
- TEL 货船
- HAT 矿船
- HAT 空间站
- 第 86 护卫队成员

如果确实必须替换，至少要提前保存：

- 当前结构百分比
- 阵营关系
- 所属编队/指挥关系
- 是否计入幸存人数

不然评分很容易直接归零，或者统计项全部丢失。

## 12. 评分系统：先算分，再出结果页

任务 2 最后评分一度显示 0 分，根本原因不是“公式算错”，而是评分数据根本没在正确时机写入结果系统。

稳定经验如下：

- 所有评分变量要在 `ScenarioFinished` 之前就准备好。
- `add_scenario_stat` 不能晚于结果页初始化。
- 如果最后还有剧情对白，结果页触发点要显式延后。
- 语言文本不要写死在 MD，统一走 `/t/`。

这次已经验证可行的评分拆法：

- 队长机结构
- TEL 货船结构
- HAT 矿船结构
- HAT 空间站结构
- 第 86 护卫队幸存数
- 击毁 Xenon 数

建议以后新增评分项时也保持这个模式：

- 每个评分项一个独立变量
- 每个显示项一个独立 `/t/` 文本
- 计算逻辑和展示逻辑分开

## 13. 击杀统计：不要假设“destroyed”一定等于“玩家击杀”

任务 2 里关于 Xenon 击杀统计已经说明一件事：

在 X4 的任务脚本里，“一条船死了”很容易监听到；  
“这条船究竟算不算玩家这边击杀”则未必稳定。

如果后续任务想继续保留击杀分，建议优先这样设计：

- 先把“敌人死亡数量”做成稳定统计。
- 再根据需求决定是否严格区分“玩家亲自击杀”“友军击杀”“剧情删除”。

如果必须做阵营归属判定，要额外验证：

- 最后一击来源是否稳定可取
- 友军主炮秒杀、大型舰火力覆盖时会不会全部算给友军
- 剧情销毁、切场景删除会不会误记为击杀

如果没有完全验证宁可降级规则，也不要保留一个看起来有、实际上常年记 0 的评分项。

## 14. 关系移交：不要太早做

第 86 护卫队从玩家阵营移交给 HAT，这类操作时机很敏感。

经验上：

- 关系移交前，护卫队最好仍归玩家，避免任务中途命令断掉。
- 如果过早移交，可能影响：
  - 玩家对其下令
  - 幸存人数统计
  - 部署流程归属
  - 后续对话逻辑

这次更合理的做法是放在 612 对话之后，再做移交。

以后凡是“剧情尾声交还控制权”的桥段，都建议晚做、不建议提前。

## 15. 地图坐标与缩比：一定要把门、航道、辐射区一起看

任务 2 的地图调整已经证明：

- X/Z 是星图平面坐标。
- Y 是垂直于星图平面的高度。
- 当我们说“地图右上方”，本质上就是 X 正方向、Z 正方向。

但真正危险的不是坐标本身，而是缩比。

如果对某个星区做 1/2 缩放，至少要同步核对：

- 星门位置
- 超级高速航道位置
- 任务锚点位置
- 空间站位置
- 辐射区范围

其中辐射区最容易出现“看起来缩了，实际不扣盾”的假成功状态。  
因此辐射区必须做一次单独实测，确认：

- 伤害/护盾扣减逻辑还在
- 范围和视觉表现匹配

## 16. BGM 控制：剧情曲和战斗曲最好按 Phase 管

任务 2 的 BGM 已经形成了一套比较稳定的经验：

- 战斗 Phase 使用交战 BGM
- 非战斗 Phase 恢复阵营/场景主题 BGM
- 特定高潮阶段单独指定曲目

比较稳的切法是：

- Phase 2 / 3：交战 BGM
- Phase 4 部署卫星：专属任务曲
- Phase 5 大规模入侵：高潮战斗曲
- Phase 8：潜入/压迫感曲目
- Phase 9 结束后：结尾主题曲，持续到任务结束

要点：

- 切曲最好绑定 Phase 入口，不要绑在模糊的对话中段。
- 如果希望战斗时仍能自动切换战斗层级，不要用过度“硬锁死”的方式覆盖全程。
- 某个曲目只想在一个 Phase 生效时，要记得在离开该 Phase 后显式换掉。

## 17. 跨星区镜头：优先假设“远处看不到”

任务 1 和任务 2 都验证了一点：

镜头能不能拍到，不只是镜头脚本问题，还和渲染范围强相关。

经验上请默认：

- 太远的舰船不会稳定渲染。
- 跨星区镜头如果没有特殊处理，不要假设一定能成功看到。
- 想让玩家看见的对象，最好要么靠近玩家可视范围，要么使用已经验证过的特殊场景做法。

也就是说：

- “10 条 I 刷在很远处再给镜头”这种方案，理论上很震撼，实战不一定看得见。
- 稳妥方案通常是把镜头目标拉进可渲染距离，或者干脆取消镜头，只保留任务层面的威胁表达。

## 18. 本地化工作流：中文先定稿，再修英文

这次任务 2 的流程已经比较成熟，推荐以后继续沿用：

1. 先把中文台词和任务文本定稿。
2. 清理 `/t/` 文件里的乱码、废弃项、未使用条目。
3. 再同步英文翻译。
4. 最后检查 MD 里是否还残留硬编码中文评分项或说明文本。

英文翻译的注意点：

- 保持 ID 完全一致。
- 保持占位符、特殊记号、换行结构一致。
- 先追求语气自然，再追求逐字直译。
- 如果中文已经明显口语化，英文也不要翻成说明书腔。

## 19. 任务收尾前的最低校验清单

任务 2 这种复杂度的战役，建议每次大改后至少做下面几项校验：

- 扩展内所有 XML 能否正常解析。
- `L086` 与 `L044` 的 ID 集合是否完全一致。
- 新增的 `{page,id}` 引用在中英文里是否都存在。
- 评分项名字是否都已经从 `/t/` 读取。
- 剧本文档里的台词是否和 `L086` 当前版本一致。
- 关键剧情物体是否在结果页前仍然存在。
- 修改了地图、gamestart、动态室内后，是否用全新存档重开测试。

其中最后一条非常重要：  
旧存档极容易让人误判“脚本没生效”，实际上只是旧状态没被正确重置。

## 20. 给任务 3 的直接复用建议

如果后面制作任务 3，建议直接复用任务 2 已经验证好的骨架：

- 开场系统提示 -> 玩家进入地图/站内流程
- 单独的对话 cue 链
- 单独的目标 cue 链
- 单独的刷船 cue 链
- 单独的命令 cue 链
- 单独的部署检测 cue 链
- 单独的评分汇总 cue 链

最值得直接沿用的模块有：

- 地图提示与剧情开场框架
- 室内对话和电梯权限控制
- 部署物检测框架
- 超时与正常完成汇合的 `Resolve` 结构
- `/t/` 化的评分显示

如果下一关也会跨多个星区、多个阵营、多个室内外场景，优先复用“结构”，不要只复制某个局部 cue。

## 21. 最后压缩成几句最有用的话

如果只记住任务 2 的五条经验，那就是：

- 一个 cue 只做一件事。
- 下一阶段永远挂在“上一段真正结束”之后。
- 部署提示物和真正完成物必须分开。
- 任何要计分的对象，都别轻易删了重建。
- 结果页出现之前，先把分数和显示项都写好。
