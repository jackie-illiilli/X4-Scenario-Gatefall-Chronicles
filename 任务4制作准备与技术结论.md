- 两地之间的门改用可用的 `props_gates_anc_gate_macro`。
- 异端的终结通往晨星和悲惨的天空的门保留为封锁门布景。
- 警惕凝视通往贫瘠海岸的门保留为封锁门布景。

这样 Phase 0 不依赖 Boron 主线 Cue，也不会出现可见但没有 `.exit` 的悬空星门。

原始门位点来源：

- 异端一侧：`参考资料/基础游戏/maps/xu_ep2_universe/sectors.xml:2369-2373`、`zones.xml:2291-2304`
- 警惕凝视一侧：`参考资料/BOR DLC/maps/xu_ep2_universe/dlc_boron_sectors.xml:535-539`、`dlc_boron_zones.xml:487-500`

### 2.3 环境保留

自定义宏不会自动命中原 `mapdefaults`，所以新地图显式复制了名称、环境、日照、资源区、音频区和原有残骸/矿区连接。

- 异端的终结日照：`1.81`，即 181%。
- 警惕凝视日照：`0.41`，即 41%。

## 4. 问题一：时间线战役能否自定义空间站模块

结论：可以。

这里需要区分三种需求。

### 4.1 自定义现成模块的组合、位置与表面装备

这是本关真正需要的功能，也是最成熟的方案。创建一个专用 construction plan 后，可以逐项指定：

- module macro；
- predecessor；
- snap connection；
- position/rotation；
- 模块炮塔、护盾、无人机等 upgrades。

官方 Timelines 直接这样做：

- 地图站点用 `<plan ref="scenario_waveattack_station"/>`：`参考资料/TimelineDLC/maps/timelines_map_waveattack_galaxy_macro.xml:22-27`
- 逐模块 plan：`参考资料/TimelineDLC/libraries/constructionplans.xml:4787-4807`
- 指定生产模块及表面装备：同文件 `6018-6058`
- plan 内包含 S/M 造船模块：同文件 `218-270`

同一 plan 可以混用 BOR、TER 和通用模块，只需要逐个核对 snap connection。

### 4.2 在 Phase 4 中途生成或升级前进基地

也可以。推荐在 Phase 4 用 MD：

```xml
<create_station
  macro="macro.station_gen_factory_base_01_macro"
  constructionplan="'gatefall_scenario_04_forward_base'"
  state="componentstate.operational"
  ... />
```

官方运行时创建站点的例子在 `参考资料/TimelineDLC/md/scenario_trading_1.xml:193-197`。还可以用 construction sequence 追加少量模块，做“基地升级”演出。

### 4.3 让玩家自由打开“计划建造”摆模块

引擎允许，但不建议放进第四关。完整流程还需要：

- 玩家所有权；
- 模块蓝图；
- build plot；
- build storage；
- builder；
- 建造材料和等待时间。

基础场景范例在 `参考资料/基础游戏/md/scenario_advanced.xml:166-172,299-326,3050-3080,3304-3356`。

Gatefall 现有关卡采用 0Cr、关闭 jobs/GOD/masstraffic 的独立场景结构，若开放完整建造链，所有建造船、材料、蓝图和 UI 引导都必须自行补齐，会严重打断战斗/RTS节奏。

最终建议：

- 创建固定的 `gatefall_scenario_04_forward_base` construction plan。
- Phase 4 直接以 operational 状态生成。
- 如果需要升级感，只做 2 至 3 个脚本化升级档位，例如维修、存储、防御或 S/M 支援能力。
- 不在本关开放完整自由空间站编辑器。

如果所谓“自定义模块”是创造全新外形或全新功能的 module macro，也能做到，但还需要 component、macro、module ware、modulegroups、modulesets 等完整资产链，本关没有必要走这条路线。

## 5. 问题二：能否创建只属于本关的指定配方

结论：可以，但推荐使用关卡局部的 MD 配方表，而不是接入完整原生造船经济。

### 5.1 原生 production method 可以做，但不是最佳方案

可以在 `libraries/wares.xml` 定义唯一 production method 和配方，并只在第四关授予蓝图。官方 Timelines 也有场景专用 ware 的先例：

- 场景专用改装件配方：`参考资料/TimelineDLC/libraries/wares.xml:259-270`
- 只在对应场景舰船上使用：`参考资料/TimelineDLC/libraries/god.xml:443-472`
- 单场景初始化授予蓝图：`参考资料/TimelineDLC/md/scenario_waveattack_antigone.xml:155-169`

但 `libraries/wares.xml` 是扩展级全局数据，不能真正按 gamestart 条件加载。只能通过以下方式做到“功能上只在本关可用”：

- 唯一 method/ware ID；
- 其他关卡不授蓝图；
- 其他关卡不生成所需资源或模块；
- 第四关保持 `nosave`，不会把状态带出场景。

更大的问题是，原生造一艘完整舰船需要同时为船体、引擎、推进器、盾、武器和炮塔提供生产方式；原生菜单也不能直接表达“2 艘九头蛇 + 4 艘梭鱼”的舰队包，更不能天然锁死指定 loadout。

### 5.2 推荐的 RTS 配方

第四关文档已经决定不接完整 X4 经济链，而使用脚本化战术资源。建议正式实现为：

```text
矿船采矿
  → 在前进基地交付矿石
  → MD 移除货舱矿石并增加 $TacticalResource
  → 玩家选择防空/反舰/矿工补充等舰队包
  → 校验并扣除资源
  → 等待该配方的制造时间
  → create_ship + 固定 loadout + 组建舰队
```

配方表只存在于 `gatefall_scenario_04` 的 MD namespace 中，每项至少保存：

- `id`
- `cost`
- `delay`
- `unit cap`
- `ship macro`
- `loadout id`
- `composition/commander relation`

这样有几个直接优势：

- 真正只属于第四关；
- 精确生成预设舰队包；
- 能强制使用本轮已经建立的 loadout；
- 能做单位上限、冷却、增援点、失败退款等 RTS 规则；
- 不需要完整 wharf、蓝图和经济物流链。

如果 UI 必须显示一种实体“战术资源”，可以以后新增隐藏 container ware，但资源兑换和舰队生成仍建议由 MD 控制。

最终决策建议：

- 空间站：固定 BOR/TER 混合 construction plan，加少量脚本化升级。
- RTS 制造：矿石交付 → MD 战术资源变量 → 预设舰队包。
- 原生 production method：保留为未来真正制作完整造船厂/经济关卡时的备选，不用于第四关首版。

## 6. 下一步制作入口

5. 再补海象、梭鱼、鲨鱼、东京、矿船等尚未明确固定配装的单位。
6. 最后接 M0、双电站机制、评分与第五关伏笔。
