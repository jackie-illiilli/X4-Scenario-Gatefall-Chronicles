# OurMod：目标与路线决策

## 当前状态

- 本文记录 2026-06-28 完成的可行性分析和路线决策。
- 隐藏菜单入口与 39 项容量压测已经完成，旧容量测试扩展已移除。
- 当前原理验证扩展为 `extensions/ourmod_demo_xenon_duel`：一个独立最小星系、一艘玩家 Asgard 与一艘 Xenon I。
- Demo 注册为无解锁条件的 `chapter="0"` 场景；摧毁 Xenon I 固定获得 5 分，对应五星，失败为 0 分。
- Demo 已通过 XML、资源引用、入口 ID/MD module、地图宏索引和评分阈值的静态链路检查，等待游戏内战斗与结算验收。
- `OurMod/参考资料` 中的文件只用于研究，不应原地修改或打包进最终 Mod。

## 最终目标

创建一个依赖 **X4: Timelines** DLC 的自定义时间线战役：

- 从官方 Timelines 主菜单入口进入官方 Hub。
- 复用官方隐藏场景控制台、场景选择、加载和结算体验。
- 制作一套由六个叙事章节组成的原创战役。
- 尽量不修改基础游戏 Lua，降低游戏更新造成的兼容性风险。
- 不替换、不阻断、不改变官方六章主线和官方隐藏任务的正常进度。

## 已选主方案：隐藏场景战役

### 基本结构

- 我们的全部场景注册为 `chapter="0"`。
- 六章由 OurMod 自己的 userdata 和 MD 脚本管理，是“叙事章节”，不是官方菜单中的引擎章节。
- 玩家不需要完成第六章终局任务“龙之焰”，即可解锁我们的战役入口和首批任务。
- 任务可以同时开放、线性解锁、分支解锁、跨章节解锁，或根据评分和剧情状态解锁。
- 未满足 `<unlock>` 条件的隐藏任务不会显示；满足后才进入隐藏任务列表。

### 入口

- 默认复用官方隐藏场景控制台及其菜单模式。
- 后续可以在 Hub 中增加一个使用官方隐藏控制台资源的物理控制台，但这只会增加入口位置，不会创建独立任务池。
- 所有隐藏控制台都会打开同一个 `chapter="0"` 场景集合。

### 进度与结算

- 所有自定义 ID 和 userdata 使用 `ourmod_` 前缀，避免与官方数据重名。
- 创建自己的战役管理脚本，负责叙事章节、任务解锁、分支和重置。
- 创建精简的 `OurMod_ScenarioFinished` 结算库，写入官方菜单需要的场景成绩数据并打开官方结算菜单。
- 自定义结算不得递增官方 `timelines_scenarios_finished`，不得触发官方 Mitsuno 通讯、官方全场景星级成就或官方排行榜上传。
- 返回 Hub 时可以使用官方 `timelines_hub_spawnmethod` 临时值；该值会被官方 Hub 读取后清除。

## 主方案的既定限制

- 官方隐藏任务和 OurMod 任务会出现在同一个隐藏任务列表中。
- 在不修改 Lua 的前提下，不能让不同物理控制台分别显示不同任务集合。
- 隐藏任务菜单不显示六个章节标签，也不显示尚未解锁的任务节点。
- 六章编号需要体现在任务名称、描述、剧情提示或我们自己的 Hub 交互中。
- 同时显示过多隐藏任务可能使列表过长；设计时应控制同一阶段的可见任务数量。
- 主方案复用完整官方 Hub，而不是删减版 Hub。

这些限制是为了换取不修改场景菜单 Lua、尽可能兼容未来游戏版本。

## 任务和章节设计原则

- 没有 `<unlock>` 的隐藏任务立即可用，不要求先完成任何前置任务。
- 多个任务可以依赖同一个前置任务，从而形成并行分支。
- `<unlock>` 中多个条件共同满足后才解锁；复杂的“或”条件由 MD 先汇总为一个 `ourmod_` userdata。
- `comparison="1"` 可表示大于等于，`comparison="0"` 表示精确等于，`comparison="-1"` 表示小于等于。
- 若使用“阶段大于等于 N”，旧任务会保留并可重玩；若使用“阶段等于 N”，进入下一阶段后旧任务会从列表消失。
- 建议每个叙事阶段同时显示约 3–6 个任务，是否保留旧任务待战役设计阶段决定。

## 已确认的官方主线特性

- 官方六章并没有“每章必须且只能有五个任务”的限制。
- 官方六章的总任务数依次为 4、5、6、6、5、6；其中每章各有一个终局任务。
- 普通任务按每行三个自动排列；终局任务单独显示。
- 官方主线章节中的普通任务通常同时开放；完成全部普通任务和所需总评分后解锁终局任务。
- 下一引擎章节依赖上一章终局任务至少获得一星。

这些主线规则不约束 `chapter="0"` 的隐藏战役。

## 备选路线

### 备选 A：在官方主线后追加第七、八章

- 数据层支持任意非负章节编号。
- 第七章会在“龙之焰”至少获得一星后解锁，第八章依赖第七章终局任务。
- 官方 Hub 的通用章节统计能识别新增章节。
- 场景菜单标题栏固定为六章，需要小型 Lua 布局修改。
- 还需隔离官方“最后一章”、Mitsuno 通讯、全星成就和全局任务计数。

当隐藏战役方案不能满足叙事或界面需求时，优先切换到此路线。

### 备选 B：独立主菜单入口和自定义精简 Hub

- 技术上可行，自由度最高。
- 需要主菜单和场景菜单 Lua 接入，长期兼容和维护成本较高。
- 当前不采用。

### 最后手段：直接替换官方 Timelines 战役

- 实现阻力较低，但会破坏玩家正常游玩官方内容的目标。
- 只有其他路线出现无法解决的引擎限制时才考虑。

## 明确不采用的方案

- 仅通过五个普通控制台中的某一个导入独立战役。

五个普通控制台只是同一场景菜单的五个物理站位。控制台接口没有“战役 ID”或任务过滤参数；增加 `console_06` 也只会增加站位，不能建立独立的 1–6 章任务池。

## 实施约束

- 最终 Mod 必须通过 `content.xml` 强制依赖 `ego_dlc_timelines`。
- 使用扩展 diff 和新增文件，不直接覆盖基础游戏或 DLC 原文件。
- 不修改 `ego_gameoptions/gameoptions.lua` 和 `menu_scenario_selection.lua`，除非正式切换路线。
- 不复用官方任务 ID、userdata、文本页或 MD 脚本名作为 OurMod 的可写状态。
- 场景脚本、地图、文本、图标和进度状态均使用统一的 `ourmod_` 命名空间。
- 参考官方场景实现时，应复制所需结构并改名，不能让自定义场景继续调用会污染官方进度的结算逻辑。

## 后续实施顺序

1. 创建扩展骨架和 Timelines 强制依赖。
2. 制作一个最小 `chapter="0"` 测试场景。
3. 验证隐藏控制台发现、加载、评分、结算和返回 Hub。
4. 验证自定义结算不会推进官方 Hub 剧情和官方成就统计。
5. 建立 `ourmod_campaign_stage`、任务标志和自定义重置机制。
6. 再开始正式六章剧情和场景制作。

## 尚待设计阶段决定

- 复用现有隐藏控制台，还是在 Hub 中增加一个额外物理入口。
- 已完成任务长期保留供重玩，还是在章节推进后隐藏。
- 六个叙事章节各自的任务数量、并行分支和评分门槛。
- 自定义战役的重置入口；官方“重置时间线进度”不应被假定会清除 `ourmod_` userdata。
- 是否向主宇宙发放奖励，以及奖励与官方 Timelines 解锁的隔离方式。
- 失败、玩家死亡、中途退出和 Mod 移除时的数据处理策略。
